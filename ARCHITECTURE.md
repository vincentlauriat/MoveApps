# Architecture

## Vue d'ensemble
`move-app.sh` est un script bash unique et autonome (compatible bash 3.2 — la version livrée par défaut sur macOS) qui migre un répertoire de projet depuis `~/Documents/Github` (iCloud Drive) vers `~/DevApps` (local) sans le casser.

## Déroulé (par projet sélectionné)
1. **Sélection** — arguments CLI, ou interactif : `fzf -m` si installé, sinon un menu numéroté avec filtre par sous-chaîne (boucle jusqu'à obtenir une sélection non vide).
2. **Détection du stack** — présence de `package.json`, `requirements.txt`/`pyproject.toml`/`Pipfile`, `*.xcodeproj`/`*.xcworkspace`, `Cargo.toml`, `go.mod`, `.git` (via des appels `find` peu coûteux, maxdepth 2).
3. **Vérification iCloud** — `find -name '*.icloud'` ; si des fichiers sont trouvés, `brctl download` est déclenché et le script attend leur matérialisation (nombre de tentatives borné).
4. **Détection des venvs Python** — `find -name pyvenv.cfg` (et non une simple correspondance de nom de dossier, pour éviter les faux positifs), car un venv n'est pas relocalisable : ses scripts `bin/*` contiennent un shebang absolu codé en dur vers le chemin du venv lui-même. Pour chaque venv trouvé, `pip freeze` est capturé dans un fichier temporaire *avant* le déplacement, tant que l'ancien chemin absolu est encore valide.
5. **Snapshot git (avant)** — branche, HEAD, nombre de fichiers modifiés, le tout encapsulé dans une fonction `with_timeout` maison (aucun binaire GNU `timeout` sur cette machine), car les opérations git sur des arborescences gérées par iCloud peuvent être étonnamment lentes.
6. **Confirmation** — le plan est affiché et l'utilisateur confirme par projet, sauf avec `--yes`. `--dry-run` s'arrête ici.
7. **Déplacement (`move_dir`)** — essai de `mv` en premier (atomique et rapide quand ça fonctionne). En cas d'échec (en pratique : `Operation timed out`, car l'extension FileProvider d'iCloud intercepte `rename()` même entre chemins sur le même volume), repli sur `ditto` (l'outil d'Apple, qui matérialise/copie correctement le contenu iCloud) suivi d'une comparaison du nombre de fichiers entre source et destination. La source n'est supprimée (`rm -rf`) que lorsque les comptes correspondent — une copie non vérifiée n'est jamais supprimée.
8. **Recréation des venvs** — dans le nouvel emplacement, le dossier du venv déplacé (désormais cassé) est supprimé, un nouveau est créé avec `python3 -m venv`, puis les paquets sont réinstallés depuis le fichier freeze capturé à l'étape 4.
9. **Réinstallation node optionnelle** — uniquement avec `--reinstall-node` : détecte un lockfile pnpm/yarn/npm et réinstalle `node_modules` de zéro. Désactivé par défaut car un simple déplacement de répertoire ne casse généralement pas `node_modules` (ses symlinks/hardlinks sont soit relatifs, soit pointent vers des stores externes fixes, par ex. le store global pnpm).
10. **Lien symbolique de compatibilité optionnel** — `--keep-symlink` recrée l'ancien chemin comme lien symbolique vers le nouveau, pour les outils/IDE qui le référencent encore.
11. **Vérification (après)** — snapshot git comparé à l'étape 5 ; `grep -rl` sur l'arborescence déplacée (en excluant `.git`, `node_modules`, `.venv`, les dossiers de build/cache) à la recherche de toute référence résiduelle à l'ancien chemin absolu littéral ; `find -type l` + `readlink` pour repérer les liens symboliques pointant désormais vers une cible absolue inexistante.
12. **Rapport** — une ligne par projet OK / AVERTISSEMENT / ECHEC, plus un tableau récapitulatif final.

## Contraintes de conception
- **bash 3.2 uniquement** — pas de `mapfile`, pas de tableaux associatifs, pas de `${var,,}`. Les sélections sont construites avec des tableaux indexés classiques et des boucles `while IFS= read -r`.
- **Aucun coreutils GNU supposé présent** — il n'y a pas de binaire `timeout`/`gtimeout` sur cette machine ; remplacé par une fonction `with_timeout` maison (job en arrière-plan + watcher).
- **Sécurité avant vitesse** — n'écrase jamais une destination existante, ne supprime jamais une source avant qu'une copie soit vérifiée, les venvs Python sont toujours reconstruits à partir d'une liste de paquets capturée plutôt que d'être conservés tels quels après un déplacement.

## Point d'attention découvert en conditions réelles
Même si `~/Documents/Github` et `~/DevApps` rapportent le même identifiant de périphérique de système de fichiers (`stat -f "%d"`), un simple `mv` entre les deux n'est pas garanti d'être un renommage de métadonnées seul — l'extension FileProvider d'iCloud Drive intercepte l'opération et peut la faire expirer. Ceci a été découvert lors du premier test réel et n'est pas un cas limite théorique ; le repli `ditto` dans `move_dir` le gère de façon transparente.

# MoveApps.app — réécriture native SwiftUI (actuelle)

Le `move-app.sh` ci-dessus est désormais un fallback CLI non maintenu, gardé comme référence. L'outil actif est **MoveApps.app**, une app macOS native qui porte la même logique en Swift, ajoute une interface (menu bar + fenêtre principale), et rend les transferts bidirectionnels.

## Pourquoi une réécriture complète plutôt qu'un wrapper autour du script
Le rollout bash est terminé (121/121 projets), et c'est désormais un outil du quotidien piloté depuis une UI sur plusieurs Macs — un appel `Process` vers le script bash aurait fonctionné, mais Swift natif apporte la concurrence structurée pour un retour de progression en temps réel, des services testables derrière des protocoles (critique pour reproduire en sécurité des modes de défaillance comme le bug de perte de données `ditto` — voir plus bas — sans jamais toucher aux vrais dossiers de projets), et un modèle `TransferWarning`/`TransferResult` typé plutôt que le parsing de texte libre en sortie du script.

## Logique portée depuis `move-app.sh` (doit reproduire exactement son comportement)
Détection de stack, matérialisation des stubs iCloud (`*.icloud`), détection de venv Python par présence de `pyvenv.cfg` (pas par nom de dossier) avec capture/recréation `pip freeze`, snapshot git avant/après (branche/HEAD/dirty-count), repli `mv`→`ditto` avec vérification avant suppression de la source, détection de symlinks cassés et cross-projet, scan des références résiduelles à l'ancien chemin absolu. Détail comportemental complet dans la section « legacy » ci-dessus.

## Décisions techniques spécifiques au port Swift
- **iCloud** : `FileManager.startDownloadingUbiquitousItem(at:)` + polling borné sur `ubiquitousItemDownloadingStatus`, derrière un protocole `ICloudMaterializing` — pas `brctl`/`NSMetadataQuery`.
- **Git & copie** : `Process` appelant `/usr/bin/git` et `/usr/bin/ditto` (même pattern que `TracerouteService` dans `~/DevApps/NetworkTools/NetCheck`), pas libgit2 — zéro divergence de comportement par rapport aux 121 exécutions validées de l'outil bash.
- **Retour de progression** : `actor TransferPipeline` exposant `AsyncStream<TransferStep>`, consommé par un view model `@MainActor @Observable`. Pas de Combine, pas de polling.
- **`TransferWarning` est un enum typé**, pas du texte libre — `gitDeletedFilesDetected` (le cas `onyx`) pilote un résultat `.critical` distinct de `.warning`, remonté comme une alerte non-fermable. Dans le script bash, ce n'était qu'une ligne noyée dans un message d'avertissement générique que Vincent devait lire attentivement pour la repérer ; le port ne doit pas reproduire cette fragilité.
- **Accès au dossier Documents** : `NSOpenPanel` (sélection explicite par l'utilisateur, defaults pré-remplis sur `~/Documents/GitHub`/`~/DevApps`) + security-scoped bookmarks persistés, plutôt que de compter sur le toggle TCC global Documents. Les bookmarks sont propres à chaque Mac et ne se synchronisent pas — chaque Mac fait sa propre sélection de racines au premier lancement.
- **Historique** : JSON simple via un `actor TransferHistoryStore`, pas SwiftData — le besoin (liste chronologique, pas de requêtes complexes) ne justifie pas la charge.
- **Distribution** : le repo est privé, donc le pattern habituel d'appcast Sparkle via `raw.githubusercontent.com` ne fonctionne pas sans authentification. Le v1 n'a pas d'auto-update réseau — DMG signé/notarisé via `Scripts/release.sh`, distribué manuellement entre les Macs de Vincent.
- **Accès Documents, révisé** : `RootPathsController` (dans `MoveAppsUI`) tente d'abord un bookmark `.withSecurityScope` puis retombe sur un bookmark simple, car l'app lance des sous-process `git`/`ditto` via `Process` — interdit sous l'App Sandbox, donc elle ne pourra jamais détenir les entitlements qu'exige un *vrai* security-scoped bookmark. La UX de persistance/reconfiguration (`NSOpenPanel`, `UserDefaults`, détection de bookmark périmé/révoqué remontée comme drapeau « à reconfigurer ») reste sinon exactement celle prévue à l'origine.
- **Le code de sortie de `du` n'est pas un signal de succès** : `DiskUsage.sizeBytes(of:)` (via `ProcessRunner`) ne doit pas se baser sur `result.didSucceed`. `du -sk` sort en code non-zéro (`1`) dès qu'il rencontre un sous-dossier illisible ou verrouillé (ex. « Resource deadlock avoided » sur certains bundles `.dSYM`/`.framework` — fréquent sur de vraies arborescences de projets), tout en imprimant un total valide sur stdout. Seul `result.timedOut` signifie qu'aucune réponse n'a été obtenue ; un code de sortie non nul avec une sortie doit quand même être parsé.

## Réouverture au clic sur l'icône Dock (2026-07-07)
`NSApplicationDelegate.applicationShouldHandleReopen(_:hasVisibleWindows:)` est le seul point d'entrée qu'un clic sur l'icône Dock atteint côté code, mais c'est de l'AppKit — aucun accès à l'action d'environnement SwiftUI `openWindow`, qui n'existe que sur `App`/`Scene`/`View`. `AppDelegate` expose `var openMainWindow: (() -> Void)?`, réglé une seule fois depuis le `.onAppear` du contenu de `Window("main")` dans `MoveAppsApp` (`appDelegate.openMainWindow = { openWindow(id: "main") }`) — le pont standard pour ce type de problème AppKit↔SwiftUI. `applicationShouldHandleReopen` l'appelle quand `hasVisibleWindows` est faux, puis active l'app.

## Fenêtre principale : bandeau de stats, recherche, pilules d'action flottantes (2026-07-07)
La fenêtre principale a gagné un bandeau (compteurs par racine + espace disque — réutilisant `DashboardViewModel`, auparavant réservé à la barre de menu, désormais aussi injecté dans la scène `Window("main")` — et une recherche par nom filtrant les deux colonnes) et les barres de lot/progression pleine largeur ont été remplacées par des « pilules » en capsule flottantes superposées en bas de la fenêtre plutôt qu'insérées dans le flux des colonnes. `ProjectScanner.scan` ne surface plus non plus les dossiers-conteneurs réellement vides en repli (seuls les dossiers non vides mais non reconnus comme projets continuent d'apparaître, conformément à son contrat d'origine « rien ne disparaît silencieusement »), et `projectGroups` (dans `MainWindowView`) trie désormais les projets isolés et les dossiers-conteneurs en une seule liste alphabétique entrelacée plutôt qu'en deux blocs triés séparément.

## Couleurs Archive/Actif — sur mesure, pas système (2026-07-07)
Archive et Actif ont chacun désormais une `Color` adaptative dédiée (`RootAccent.swift` : un ambre désaturé pour Archive, un teal désaturé pour Actif, tous deux définis comme des `NSColor` dynamiques qui changent de teinte entre clair/sombre) plutôt que `.orange`/`Color.accentColor`. C'était un choix délibéré de fidélité à un mockup de design HTML approuvé : `.orange` et la couleur d'accent système de Vincent sont plus saturés que la paire désaturée du mockup, et utiliser l'accent *système* spécifiquement pour « Actif » ferait dépendre son intensité du choix d'accent personnel de Vincent plutôt que de l'identité à deux tons pensée pour cet écran.

## Génération de l'index des projets (`INDEX.md`)
`IndexGenerator` (MoveAppsCore, une `struct` `Sendable` pure) construit un index Markdown unifié couvrant les **deux** racines et en écrit une **copie identique dans chacune** (`<active>/INDEX.md` et `<archive>/INDEX.md`) — c'est la réponse de l'app à « garder un catalogue des projets dans les deux répertoires, généré par MoveApps lui-même ». Il réutilise `ProjectScanner` pour énumérer les projets réellement transférables de chaque racine (donc il voit exactement la même liste que la UI de transfert, dossiers-conteneurs décompactés), les groupe par racine → dossier-catégorie → niveau racine, et rend `**nom** — \`chemin\` — stack — description` par entrée. Les descriptions sont **extraites du `README.md`** de chaque projet (première vraie ligne de prose, avec un filtre robuste qui écarte titres, badges, images, HTML/attributs bruts, bannières ASCII art, barres de navigation multilingues et commandes shell/installation, décode les entités HTML, et retombe sur le titre H1). La taille disque par projet est volontairement omise (un `du` par projet serait trop lent pour une action « régénérer à chaque transfert »). Déclenché de deux façons : le bouton « Régénérer l'index » du tableau de bord barre de menu (`DashboardViewModel.regenerateIndex`, avec bandeau de succès/échec inline) et automatiquement à la fin de chaque transfert/lot (`MainWindowViewModel.regenerateIndex`, fire-and-forget hors du main actor). Les deux racines étant identiques, certains liens relatifs ne résolvent que depuis la copie de leur propre racine — l'emplacement est donc rendu en chemin brut `\`conteneur/nom\`` plutôt qu'en lien cliquable, pour éviter les liens cassés d'une copie à l'autre.

## Stratégie de test pour le bug de perte de données silencieuse façon `onyx`
`DirectoryCopying` est un protocole ; la prod utilise `DittoCopier` (encapsule `/usr/bin/ditto`), les tests utilisent un `FaultInjectingCopier` qui copie fichier par fichier sur un vrai dépôt git de fixture local (`git init`/`add`/`commit` via `Process`, pas un git mocké) tout en droppant délibérément un fichier tracké tout en gardant le nombre total de fichiers égal — reproduisant exactement la forme de l'incident réel `onyx` (« le comptage matchait, mais un fichier tracké a disparu »). Le pipeline doit détecter ce cas via le diff du dirty-count git (une entrée `D `), le classer `.critical`, et — invariant de sécurité critique — **ne pas supprimer la source**.

## Verrou de prise (checkout) multi-Mac de l'Archive + taille par projet (2026-07-12)
La racine Archive est partagée (iCloud Drive) entre les Macs de Vincent, mais jusqu'ici un projet sorti de l'Archive (Archive→Actif) disparaissait purement et simplement — un autre Mac ne pouvait pas distinguer « pris par quelqu'un » de « n'a jamais existé ». `CheckoutReference`/`CheckoutReferenceStore` (`MoveAppsCore`) comblent ce manque : prendre un projet laisse un marqueur à son emplacement d'origine dans l'Archive, indiquant quel Mac l'a pris et quel jour, transformant le slot en référence plutôt qu'en projet complet. `TransferPipeline` refuse catégoriquement (blocage dur, sans possibilité de forcer) tout plan visant un projet portant déjà une référence de prise, et `ProjectScanner` détecte le marqueur avant sa logique normale de détection de projet (au niveau racine comme imbriqué dans un dossier-conteneur), afin qu'un slot verrouillé apparaisse comme tel plutôt que d'être scanné comme un vrai projet ou de disparaître silencieusement comme un dossier vide. Rendre un projet (Actif→Archive) efface le marqueur à la destination et balaie les marqueurs orphelins classés sous un autre dossier-conteneur que celui d'origine (`clearOrphans`). Un bouton « Libérer » dans la fenêtre principale efface manuellement un marqueur périmé ou erroné, derrière une confirmation, sans toucher à aucun contenu réel.

**Risque d'éviction de contenu iCloud, et pourquoi le nom du marqueur porte les faits essentiels.** Un simple marqueur JSON caché aurait eu un défaut fatal : l'Archive est sur iCloud Drive, et le *contenu* d'un fichier (pas son entrée de répertoire) peut être évincé localement, apparaissant comme un placeholder `.nom.icloud` dont le corps se lit comme vide — un Mac le lisant verrait un slot « vide » et reprendrait un projet déjà pris, exactement le mode de défaillance que la fonctionnalité doit empêcher. Ce risque a été identifié via une revue par l'advisor avant l'implémentation, pas découvert après coup. La correction : le **nom de fichier** du marqueur — `MOVEAPPS-CHECKOUT__<host-assaini>__<yyyy-MM-dd>.json` — encode directement l'hôte et le jour, car les noms survivent à l'éviction (seulement enveloppés en `.<nom>.icloud`, toujours analysable via une regex acceptant les deux formes : `^\.?MOVEAPPS-CHECKOUT__([A-Za-z0-9-]+)__(\d{4}-\d{2}-\d{2})\.json(\.icloud)?$`). Le corps JSON n'ajoute que des champs bonus (`destinationPath`, `sizeBytes`) qui retombent sur `nil` s'ils ne sont pas encore matérialisés. La lecture du dossier du marqueur ne doit pas utiliser `.skipsHiddenFiles` — le placeholder évincé est préfixé d'un point même si le vrai fichier ne l'est pas. `CheckoutReferenceStoreTests` construit un vrai fichier placeholder suffixé `.icloud` sur disque pour le prouver, plutôt que de l'affirmer comme une hypothèse.

Nom d'hôte : `Host.current().localizedName` (repli sur `ProcessInfo.processInfo.hostName` débarrassé d'un éventuel `.local`). L'agent ayant implémenté la fonctionnalité a découvert, en testant sur la machine réelle avant le commit, que `ProcessInfo.hostName` seul résout vers une chaîne de reverse-DNS du FAI sur le réseau réel de Vincent (`...[redacted-ISP-hostname]`) plutôt que vers quoi que ce soit identifiant le Mac — ce qui aurait silencieusement vidé le marqueur de son sens. Détecté et corrigé, pas livré tel quel.

La taille disque par projet — volontairement omise d'`IndexGenerator` à sa sortie initiale (voir plus haut) car un `du` par projet était trop lent pour le chemin de régénération automatique après transfert — s'affiche désormais sur chaque ligne dans la fenêtre principale et dans `INDEX.md`, sans réintroduire ce coût : `ProjectSizeCache` (`MoveAppsCore`, un acteur en mémoire uniquement, volontairement non persisté sur disque) est alimenté par `MainWindowViewModel` en tâche de fond avec une concurrence bornée (~6 `du` en parallèle) après que le scan a déjà rendu la liste instantanément, et `IndexGenerator.write` prend un paramètre optionnel de recherche de taille — le chemin de régénération automatique après transfert passe l'instantané courant du cache (best-effort, retombe sur `—` pour tout ce qui n'est pas encore mesuré) plutôt que de calculer quoi que ce soit à froid.

## Mise à jour automatique via Sparkle, et l'arbitrage repo privé/public (2026-07-12)
La v1 est sortie sans auto-update réseau car le pattern d'appcast standard de Sparkle — une lecture HTTP non authentifiée d'`appcast.xml` et du DMG, typiquement servie directement par `raw.githubusercontent.com`/GitHub Releases — ne fonctionne que contre un repo **public**, et `MoveApps` était privé. Intégrer un token GitHub dans l'app pour authentifier un flux privé a été envisagé puis écarté : c'est un secret extractible d'un binaire distribué, et ce n'est pas l'usage prévu de Sparkle. Face aux vraies options (rendre le repo source public / héberger seulement l'appcast+DMG sur une surface publique séparée en gardant le source privé / utiliser le site Pages déjà existant `vincentlauriat.github.io`), Vincent a choisi de rendre `vincentlauriat/MoveApps` public — le même montage que `MarkdownViewer` utilise déjà avec succès, d'où un câblage ici qui reprend de près son vrai `MarkdownViewerApp.swift` et `project.yml` (lus, pas devinés).

`SPUStandardUpdaterController` (construit une fois dans `MoveAppsApp.swift`) est délibérément configuré avec `automaticallyChecksForUpdates = true` mais `automaticallyDownloadsUpdates = false` — vérification en arrière-plan uniquement, l'installation exige toujours une confirmation explicite via « Rechercher les mises à jour… ». Ce n'est pas qu'une copie de `MarkdownViewer` ; cela compte davantage ici, car MoveApps lance `git`/`ditto` en sous-processus pendant un transfert via `TransferPipeline`, et un `SIGKILL` silencieux de Sparkle suivi d'un remplacement de l'app en cours d'exécution serait un mode de défaillance bien plus grave que dans un simple visualiseur de documents.

Un compte trousseau EdDSA Sparkle dédié, `MoveApps` (distinct de celui de `MarkdownViewer` — une seule paire de clés pourrait en principe servir plusieurs apps, mais des comptes par app limitent le rayon d'impact d'une clé perdue ou compromise à une seule app), a été généré une fois via `.sparkle-tools/bin/generate_keys --account MoveApps` et ne doit **jamais être régénéré** — cela ferait rejeter définitivement les futures mises à jour par toutes les copies déjà installées (cet incident précis est déjà arrivé une fois à `MarkdownViewer`, forçant un retéléchargement manuel pour tout le monde en ≤ v0.8.0 ; voir l'en-tête de son `release-full.sh` pour l'histoire complète). `Scripts/release.sh` signe désormais le DMG avec Sparkle et écrit `appcast.xml` à la racine du dépôt en toute fin de pipeline, reprenant la section équivalente de `release-full.sh`.

Un vrai bug intercepté avant qu'il ne morde : `CFBundleVersion` — le numéro de build que le comparateur de Sparkle lit réellement, pas `MARKETING_VERSION` — était resté figé à `"1"` depuis `0.1.0` et `0.2.0`. Sans conséquence tant qu'aucun flux Sparkle n'existait, mais deux releases partageant le même numéro de build sont indiscernables pour le comparateur de versions de Sparkle, ce qui aurait cassé silencieusement la détection de mise à jour dès l'activation de l'auto-update. Passé à `"2"` pour `0.3.0` ; chaque future release doit aussi incrémenter `CURRENT_PROJECT_VERSION` dans `project.yml`, pas seulement la version marketing — désormais rappelé explicitement dans le commentaire d'en-tête de `Scripts/release.sh`.
