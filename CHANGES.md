## 2026-07-01

### Added
- `move-app.sh`: interactive bash 3.2-compatible tool to migrate projects from `~/Documents/Github` (iCloud Drive) to `~/DevApps`, one or several at a time.
  - Interactive picker (fzf if available, else substring-filter menu) or direct `move-app.sh NAME...` invocation.
  - Stack detection (git/node/python/xcode/rust/go).
  - Forces download of un-materialized iCloud stub files (`*.icloud`) before moving.
  - Detects Python venvs via `pyvenv.cfg` (not just folder name), captures `pip freeze` before the move, recreates the venv and reinstalls packages after (venvs bake absolute shebang paths and are not relocatable as-is).
  - Post-move verification: git branch/HEAD/dirty-file-count comparison, scan for leftover references to the old absolute path, broken symlink detection.
  - `--dry-run`, `--yes`, `--keep-symlink`, `--reinstall-node`, `--list` flags.

### Fixed
- `mv` alone fails with `Operation timed out` when the source is inside iCloud Drive, even though source and destination report the same filesystem device ID (iCloud's FileProvider intercepts `rename()`). Fixed by adding a fallback path in `move_dir()`: try `mv` first, then `ditto` (Apple's iCloud-aware copy tool) + file-count verification, and only `rm -rf` the source once the copy is confirmed complete.

### Decisions
- `node_modules`, `target` (Rust), `.build`/`DerivedData` (Xcode) are left untouched by default on move — they generally survive a same-machine move without breaking, and forcing a reinstall/rebuild for all 126 projects would be slow for no benefit. `--reinstall-node` opts into a clean reinstall when wanted.
- Python venvs are always destroyed and recreated post-move (no opt-out) since they are guaranteed broken otherwise (hardcoded shebang paths in `venv/bin/*`).
- The tool never overwrites an existing destination and never deletes a source before the copy is verified complete — safety over speed.

### Fixed (Phase 1 build issues, caught during independent implementation)
- `Process.waitUntilExit()` spins a private `CFRunLoop` that races with concurrent pipe reads under Swift 6 strict concurrency and can hang intermittently (diagnosed via `sample` on a hanging xctest run). `ProcessRunner` now uses `process.terminationHandler` + `DispatchGroup` instead.
- `FileManager` isn't `Sendable` — services expose it as a computed `.default` property rather than an injected stored instance, to stay `Sendable` without `@unchecked`.

### Decisions (Phase 1)
- **Source deletion is deferred past the post-move git check** — a deliberate deviation from `move-app.sh`, which deleted the source inside `move_dir` before the git verification. `TransferPipeline` takes the git snapshot *after* the move but *before* any source deletion, so a newly-appeared `D ` entry (the `onyx` signature) escalates to `.critical` and the source is preserved even though the file-count check passed. Native rename is atomic (no partial-loss risk) so it deletes the source immediately; only the copy/ditto fallback path defers.
- `SymlinkVerifier` excludes `build`/`DerivedData`/`.build`/`target` from cross-project-symlink detection so self-referential build symlinks never surface as false positives (the bash script only excluded `.build` from its residual-path grep, not its symlink scan — the Swift port tightens this per the original design intent).
- `TransferPipeline` takes a `RootLocations` (a `Sendable` struct snapshot) rather than the `@Observable RootPathsSettings` object directly, since `@Observable` types aren't `Sendable`.

### Added (template post-copy init script, 2026-07-03)
- "New project from template" now runs a template's optional post-copy init script. When a copied template ships `Scripts/bootstrap.sh` (relative path exposed as `templateInitScriptRelativePath`) and the user opts in, `TemplateService.createProject` executes it via the new `InitScriptRunning` protocol (real impl `BootstrapScriptRunner` runs `bash <script> <displayName> <slug>` with Homebrew bindirs prepended to `PATH`, since a Finder-launched app inherits a minimal `PATH` without `xcodegen`/`swift`). This lets AppKitTemplate rename identifiers, regenerate its `.xcodeproj`, generate an icon and seed git in one step — the "create → ready" flow.
- `ProjectCreationResult.created` gained an `initScript: InitScriptOutcome` (`.none`/`.skipped`/`.ran`/`.failed`); when the script runs it owns git, so no separate `git init` is attempted. `TemplateService.slug(from:)` derives a space-free Xcode-friendly slug from the project name (`"My New App"` → `"MyNewApp"`).
- `NewProjectView`: a "Exécuter le script d'initialisation du modèle" toggle appears only for templates that ship a script; the result banner reports the script outcome (created & initialized / created but script failed).
