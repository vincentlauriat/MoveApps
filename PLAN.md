### Phase 6 — Archive checkout lock (multi-Mac) + per-project size (current focus, planned 2026-07-12)

**Goal** (Vincent's framing): turn the Archive root into a real shared referential across his Macs. A project living in the Archive is either **fully present** (available to take) or **just a reference** (currently checked out on some Mac) — never both. Taking a project must leave a trace (which Mac, which day) so another Mac can't take/search it by mistake. Every project row must show its own disk size, not only a per-root global total.

Design decisions confirmed with Vincent (2026-07-12): double check-out is a **hard block** (not a warn-and-override), size shows in **both** the main window and `INDEX.md` via a **background/cached** computation (no `du` added to the hot path), and a manual **"Libérer"** action exists in the UI to clear a stale/erroneous checkout trace.

**Key risk found before implementing** (via advisor review): the Archive root is iCloud Drive–backed. A *hidden* marker file's **content** can be evicted locally and appear as a `.name.icloud` placeholder — reading it would then see nothing, silently defeating the entire feature (Mac B would see an "empty" slot and re-take a project Mac A already has). Directory and file **names** survive eviction (only content does not, and eviction wraps the name as `.<name>.icloud`, which is still parseable). So the checkout marker encodes the essential facts (hostname, date) **in its filename**, not only in JSON content — content is an enhancement (adds `destinationPath`, `sizeBytes`) that degrades gracefully to "—" if not yet materialized, but "taken by whom, since when" never depends on file content being downloaded.

**Design**
- `CheckoutReference` (new model, `MoveAppsCore/Models`): `hostName: String`, `takenAt: Date`, `destinationPath: String?`, `sizeBytes: Int64?`.
- `CheckoutReferenceStore` (new service): writes/reads/clears the marker. Marker filename pattern: `MOVEAPPS-CHECKOUT__<sanitized-host>__<yyyy-MM-dd>.json`, placed inside the project's **original, unrenamed** directory (so the checked-out project keeps occupying its exact old path/name in the Archive — nothing else changes shape). Detection regex accepts both the plain form and its iCloud-evicted placeholder form (`^\.?MOVEAPPS-CHECKOUT__([A-Za-z0-9-]+)__(\d{4}-\d{2}-\d{2})\.json(\.icloud)?$`), matched via a directory listing that does **not** use `.skipsHiddenFiles` (the evicted placeholder is dot-prefixed even though the real file isn't) — this is the same nuance `ProjectScanner` already has to account for elsewhere in the codebase for iCloud stubs.
  - `clearOrphans(named:under:)`: sweeps the archive root (top level + one level into container folders, mirroring `ProjectScanner`'s own shallow structure) for a stub matching a project name, for the case where a project is checked back in under a *different* container than the one it was taken from.
- `ProjectScanner`: before running its normal `isProjectRoot`/decomposition logic on **any** directory (top-level or inside a container), checks for a checkout marker first (via a dedicated non-hidden-skipping listing) and — if found — emits a `ProjectCandidate` with `checkoutReference` set instead of scanning it as a real project. This check is load-bearing, not an optimization: a stub folder whose only child is (in evicted form) a dot-prefixed name would otherwise look empty to the existing `.skipsHiddenFiles` scan and silently vanish from the list entirely.
- `ProjectCandidate` gains `checkoutReference: CheckoutReference?` (`nil` for normal projects).
- `TransferPipeline`:
  - Archive → Active leg: once the source is confirmed gone (`sourceDeleted && !isCritical` — the existing onyx invariant, unchanged), measure the destination's size (`DiskUsage`, one `du` for the single project being transferred — cheap) and write the checkout marker back at the original Archive path instead of leaving it simply deleted. A new `TransferWarning.checkoutReferenceWriteFailed(reason:)` surfaces (escalating to `.warning`, never silent) if the marker can't be written — the exact "taken by mistake" hole reopens if this fails quietly.
  - Active → Archive leg ("check-in"): before the existing "destination already exists" guard, clear any checkout marker sitting at the destination path, *and* sweep for an orphaned marker elsewhere in the archive under the same project name (`clearOrphans`) in case the project is being returned under a different category folder than it was taken from.
  - Defensive guard: a plan whose `project.checkoutReference != nil` is refused immediately (`.failed`) without touching the filesystem — belt-and-suspenders under the UI-level hard block.
  - New `TransferStep.measuringSize` (+ French label in `ProjectListing.describe`).
- `ProjectSizeCache` (new, in-memory only — deliberately **not** persisted to disk, per Vincent's scope call): an actor holding `[URL: Int64]`, shared between `MainWindowViewModel` and wherever `IndexGenerator.write` is invoked, constructed once in `MoveAppsApp.swift`. `MainWindowViewModel.refresh()` renders the project list immediately (as today, no `du` calls) then kicks a detached background task computing sizes with bounded concurrency (~4–8 parallel `du`s via `DiskUsage`), warming the cache and updating `projects` incrementally as results land.
- `IndexGenerator.write`: gains an optional size lookup (backed by `ProjectSizeCache`) and renders a size column per line; falls back to `—` when a project isn't in the cache yet (e.g. right after a fresh transfer) rather than computing cold — preserves the existing "no `du` on the auto-regenerate-after-transfer hot path" guarantee. The dashboard's manual "Régénérer l'index" button may compute cold since it's an explicit, infrequent user action.
- UI (`MainWindowView`): a locked Archive row shows a lock badge, "Pris par `<host>` le `<date>`", a disabled transfer affordance, and a "Libérer" button behind a confirmation dialog (explicitly warns this only clears the *trace*, not any real content, and should only be used when certain nobody else is using the project) calling `CheckoutReferenceStore.clear` → full removal of the stub's marker (and the now-empty directory) so the slot cleanly disappears until manually restored. Every row (locked or not) gains a size label using the existing (until now unused) `ProjectCandidate.sizeBytes` + `ByteFormat`.

**Tests**: `CheckoutReferenceStoreTests` (write/read/clear round trip, both marker states — plain and iCloud-evicted-placeholder — parse correctly), `ProjectScannerTests` (stub recognized top-level and nested in a container, doesn't get scanned as a normal project), `TransferPipelineTests` (Archive→Active writes a correct stub; Active→Archive clears it and round-trips; clears an orphaned stub in a different container; refuses a plan against an already-checked-out project without touching disk; the onyx critical path still does **not** write a stub, preserving the existing safety invariant) — extends the existing `FaultInjectingCopier` fixture to cover both new branches. `IndexGeneratorTests` (size column present when cached, `—` when not).

**Not doing (explicitly out of scope, per Vincent's answers)**: warn-and-override on double check-out (hard block only), disk-persisted size cache across app launches (in-memory only, rebuilt each session), automatic size recompute on every transfer (background/cached only).
